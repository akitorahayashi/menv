## Focus

Define a narrow observer role boundary that produces reusable analytical signals from repository evidence.

## Analysis Points

- Scope boundary precision: one stable analytical concern, no overlap with implementation planning.
- Signal class quality: recurring patterns instead of one-off incident descriptions.
- Evidence contract clarity: minimum proof for each claim and how to reject weak claims.
- Language precision: concrete and falsifiable wording without role ambiguity.

## First Principles

- A role definition exists to constrain decision quality, not to list tasks.
- Reusable analytical rules outperform repository-specific checklists.
- Scope rejection criteria are mandatory; broad roles degrade output reliability.
- Evidence standards must be explicit enough to fail unverifiable claims.

## Guiding Questions

- What analytical boundary does this role own that another role should not own?
- Are analysis points reusable signal classes rather than examples or action plans?
- Can each requirement be validated directly from repository artifacts?
- Does the wording prevent drifting into implementer or innovator duties?

## Anti-Patterns

- Role scope broad enough to absorb unrelated analytical domains.
- Analysis points written as implementation suggestions or refactoring plans.
- Repository-specific path checklists presented as reusable analytical methods.
- Evidence requirements missing, weak, or impossible to falsify.

## Evidence Expectations

- Cite direct file/line evidence when asserting structure or quality constraints.
- Differentiate observed facts from interpretation and state confidence explicitly.
- Reject claims that cannot be validated from current repository state.
- Prefer minimal, high-signal evidence over verbose but low-value citation lists.
